# 🔧 Hermes Skill 系统升级方案：从"采集合集"到"内容工厂"

> 调研时间：2026-04-28  
> 当前状态：已完成全量调研

---

## 一、现状诊断

经过对 Hermes 的 74 个技能(skills)、config.yaml、hotspot-engine 脚本、
cron 任务、SOUL 身份文件的完整调研，得出以下结论：

### ✅ 已存在的优势

| 能力 | 状态 | 评价 |
|------|------|------|
| 热点采集自动化 | ✅ 已落地 | 每日+每周定时，10个源，128条/daily |
| 去重指纹引擎 | ✅ 已落地 | 指纹预检+后检，TTL过期 |
| Skill 体系结构 | ✅ 完善 | 18个分类，74个技能，目录规范 |
| SOUL 人格身份 | ✅ 已落地 | 完整身份文档，标签体系 |
| Subagent 并行执行 | ✅ 已有 skill | 可独立调度子任务 |
| 内容模板体系 | ✅ 已有 | report_template.md + 参考文件 |
| 定时任务调度 | ✅ 已落地 | 每日8点/每周一8点 |

### ⚠️ 存在的缺口（痛点）

| # | 缺口 | 具体表现 | 严重程度 |
|---|------|---------|---------|
| 1 | **热点→内容转化断裂** | 采集报告（128条原始数据）到最终内容产出之间，没有任何AI辅助环节。你要自己看报告→选题→构思→写稿 | 🔴 高 |
| 2 | **缺少RIVET内容生成技能** | SOUL 定义了 RIVET 内容框架，但没有对应的 skill 来调用它 | 🔴 高 |
| 3 | **单稿多平台适配缺失** | 同一选题要手动适配抖音/小红书/B站三套风格和长度 | 🔴 高 |
| 4 | **选题评估依赖直觉** | 128条数据中哪些值得追、优先级如何，没有结构化判断 | 🟡 中 |
| 5 | **脚本风格不一致检查缺失** | 长期产出无法自动校对人设一致性（语气/密度/立场） | 🟡 中 |
| 6 | **创意库/选题池为空** | 没有积累"暂不追但值得记"的半成品选题库 | 🟢 低 |
| 7 | **无法做竞品选题复盘** | 不知道同行最近在追什么、什么火了 | 🟢 低 |
| 8 | **36氪数据源有回填旧数据Bug** | 持续产出2020年旧数据 | 🟢 低 |

## 二、核心思路：补齐产线

```
当前状态：
  [采集] → 128条原始数据 → 你手动选→你手动写→你手动改平台

目标状态：
  [采集] → [选题评估] → [内容骨架(RIVET)] → [多平台适配] → [人设审核]
              ↓
          选题池存档
```

用 **4 个新 skill** 补齐这个链条，每个 skill 都设计为：
- 可以被单独调用（你主动用）
- 可以被链式调用（自动化产线）
- 可以复用于已存的热点报告（自由选内容）

## 三、新增 Skill 方案

### Skill A: `content-strategy` — 选题评估与策略引擎

**解决的问题**：从热点报告（128条）中自动筛选出值得你追的选题，并给出优先级和切入角度。

**输入**：热点报告路径 或 人工输入的某个话题  
**输出**：结构化选题评估单（优先级、角度、适配人设、风险提示）

**核心逻辑**：
1. 将每条热点与 SOUL 身份核心论点（"真实稳定的自我"）匹配度打分
2. 与三个受众画像（转型者/探索者/觉醒者）匹配度打分
3. 与当前平台（抖音/小红书/B站）匹配度打分
4. 结合时间敏感度（热点是否有时效性）综合排序

**典型用法**：
```
你：帮我从今天的热点报告中抽Top5选题
我：使用 content-strategy skill → 输出5个选题 + 每个的人设适用性评估
```

---

### Skill B: `rivet-content-generator` — RIVET 内容骨架生成器

**解决的问题**：把选定的"话题"转化成符合 SOUL 风格的完整内容骨架（脚本大纲）。

**输入**：话题描述 + 目标平台（抖音/小红书/B站）  
**输出**：RIVET 五段式内容骨架

**核心逻辑**：

| 阶段 | 产出 | 设计原则 |
|------|------|---------|
| R - Rupture | 反常识钩子（第一句/标题） | 打破认知平衡 |
| I - Illuminate | 核心洞察+底层结构 | 指出受众看不到的东西 |
| V - Validate | 数据/案例/类比支撑 | 每个论据都在做事 |
| E - Embody | 具身化类比/个人故事 | 抽象概念变可感知 |
| T - Transform | 处于ZPD内的下一步 | 刚好够得着 |

**平台差异自动适配**：
- 抖音（60-180s）：Ru 前3秒定生死，I/V 压缩，E/T 简短
- 小红书（图文）：R 标题，I 正文前段，V 列表/对比，E 个人体验，T 行动清单
- B站（10min+）：R完整引入，I/V深度展开，E/T有节奏感

**典型用法**：
```
你：用RIVET生成一个B站视频骨架，讲"Microsoft与OpenAI终止独家合作"
我：输出完整RIVET骨架 + 适配B站10分钟深度的版本
```

---

### Skill C: `content-adapter` — 跨平台内容适配器

**解决的问题**：写好一个平台的内容后，自动适配到其他平台。

**输入**：一篇已写好的内容（抖音脚本/小红书笔记/B站大纲）  
**输出**：另外两个平台的适配版本

**核心逻辑**：
1. 解析输入内容的完整叙事结构
2. 提取核心论点
3. 按目标平台的风格规则重新组织：
   - 抖音 → 更口语化、更短句、前3秒最强钩子
   - 小红书 → 标题+正文+标签结构，高情绪密度
   - B站 → 有节奏感的长叙事，前30秒建立信任

**典型用法**：
```
你：已写好B站脚本，帮我转成小红书笔记
我：输出小红书版本（标题+正文+标签）
```

---

### Skill D: `voice-keeper` — 人设一致性审核器

**解决的问题**：确保每篇产出的内容"听起来还是我"——不偏离 SOUL 人格定位。

**输入**：一篇内容（脚本/笔记/大纲）  
**输出**：人设一致性报告（保留/调整/修改 三级）

**审核维度**：

| 维度 | 规则 | 偏离信号 |
|------|------|---------|
| 语气 | 平等、直接、有温度 | 高高在上的"导师语气" |
| 密度 | 高信息密度，不注水 | 废话/车轱辘话/情绪堆砌 |
| 立场 | 反常识、不贩卖焦虑 | 随大流、焦虑敲打、口号 |
| 脆弱 | 展示不确定性 | 全知全能、标准答案 |
| 提问优先 | 苏格拉底式引导 | 直接给标准答案 |

**典型用法**：
```
你：帮我审一下这篇脚本的人设一致性
我：输出审核报告 → "保持"部分 + "调整"建议
```

---

### 可选 Skill E (后期扩展): `idea-vault` — 选题池管理器

**解决的问题**：把"现在不追但以后值得做的"选题存储起来，形成素材库。

**输入**：话题描述 + 为什么值得存 + 最佳发布时机估计  
**输出**：结构化的选题记录（存入本地JSON文件）

**核心逻辑**：
1. 每次内容采集后，提出"可归档"的半成品选题
2. 按"时效敏感性"分类（高→必须追/中→一周内可做/低→永远有价值）
3. 按标签索引，支持搜索和随机抽取

## 四、执行路径设计

### 方案一：逐步落地（推荐）

| 阶段 | 内容 | 耗时估算 | 优先级 |
|------|------|---------|-------|
| **Phase 1** | 创建 `content-strategy` skill | ~2h | 🔴 P0 |
| **Phase 2** | 创建 `rivet-content-generator` skill | ~3h | 🔴 P0 |
| **Phase 3** | 创建 `content-adapter` skill | ~2h | 🟡 P1 |
| **Phase 4** | 创建 `voice-keeper` skill | ~1.5h | 🟡 P1 |
| **Phase 5** | 修复36氪旧数据问题 | ~0.5h | 🟢 P2 |
| **Phase 6** | 创建 `idea-vault` skill（选做） | ~1.5h | 🟢 P2 |

**为什么 P0 是 content-strategy + rivet-content-generator**：
- 这两个补齐了从"数据"到"内容"的核心断点
- 有这两个，你就可以做到：**"帮我从今天的报告里选最好的选题，直接输出B站脚本骨架"**
- 这是内容生产从"单兵作战"到"流水线"的关键一步

### 方案二：全部打包（一次性交付）

如果你想要一锤子买卖，可以一次性创建 5 个 skill + 修复 bug，总耗时约 10h。  
使用 `subagent-driven-development` 并行执行。

**但建议先走 Phase 1+2**——这两个最核心，体验了效果再决定要不要继续扩展。

## 五、与现有系统的集成关系

```
                上游                      产线                             下游
┌─────────────────┐    ┌─────────────────────────────────────────┐   ┌─────────┐
│  hotspot-engine  │    │ content-strategy                        │   │ 抖音    │
│  (每日/每周采集) │───→│  → rivet-content-generator              │──→│ 小红书   │
│  128条/daily     │    │  → content-adapter                     │   │ B站     │
│  51条/weekly     │    │  → voice-keeper (全流程最后审核)        │   │ 视频号  │
└─────────────────┘    └─────────────────────────────────────────┘   └─────────┘
                                │
                           idea-vault
                           (半成品选题存档)
```

### 集成细节

1. **hotspot-engine 不改动** — 保持现有采集逻辑
2. **content-strategy** 读取 `report_daily.md` 作为输入
3. **rivet-content-generator** 复用 SOUL.md 的身份定义
4. **content-adapter** 输出格式适配各平台
5. **voice-keeper** 作为最终审核节点
6. **idea-vault** 独立存储，不阻塞产线

## 六、每个 skill 的文件结构

每个 skill 遵循 Hermes 的标准目录结构：

```
~/.hermes/skills/{category}/{skill-name}/
├── SKILL.md          # 技能主文件（prompt + 使用说明）
├── references/       # 参考文件
│   └── config.yaml   # 可选：各 skill 的自定义配置
├── templates/        # 输出模板
│   └── output_xxx.md # 结构化输出示例
└── scripts/          # 可选：自动化脚本
    └── xxx.py
```

- `content-strategy` → `~/.hermes/skills/research/content-strategy/`
- `rivet-content-generator` → `~/.hermes/skills/creative/rivet-content-generator/`
- `content-adapter` → `~/.hermes/skills/creative/content-adapter/`
- `voice-keeper` → `~/.hermes/skills/creative/voice-keeper/`
- `idea-vault` → `~/.hermes/skills/research/idea-vault/`

## 七、风险和注意事项

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|-------|------|---------|
| 生成的RIVET骨架太模板化 | 中 | 低 | voice-keeper 审核 + 你人工修正 |
| 内容适配时丢失核心论点 | 低 | 高 | adapter 保留"不可删除的核心三段" |
| 选题评估过度依赖匹配度打分 | 中 | 中 | 最终由你拍板，AI只做筛选 |
| 人设审核过度保守/太松 | 低 | 低 | 可调阈值 |
| "虚拟团队"使用不顺手 | 低 | 高 | 设计为"可选调用"而非"强制产线" |
| 36氪旧数据持续污染选题评估 | 中 | 低 | Phase 5 单独修复 |

## 八、成功标准

当以下场景可以顺畅发生时，方案即成功：

1. ✅ **场景1**："帮我从今天的报告里选选题" → 5个结构化选题在30秒内输出
2. ✅ **场景2**："用RIVET写一个小红书的笔记骨架，话题是... " → 3步内完成
3. ✅ **场景3**："我把B站脚本写好了，转成抖音版本" → 自动适配
4. ✅ **场景4**：写完所有内容后跑一次"人设审核" → 挑出3-5处偏离
5. ✅ **场景5**：每周看一眼"选题池" → 发现有5个存货可以现在追

## 九、下一步行动建议

1. **先执行 Phase 1（content-strategy）**——这是最简单的skill，起步快
2. 体验一次：用今天的热点报告跑一次选题评估
3. 如果舒服 → 继续 Phase 2（rivet-content-generator）
4. 体验一次：立刻生成一篇完整内容骨架
5. 确认方向正确 → 推进 Phase 3-6

**启动方式**：确认后，使用 `subagent-driven-development` skill，每个Phase独立一个subagent任务，并行阶段可同时推进。
